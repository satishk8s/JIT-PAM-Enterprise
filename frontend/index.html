Database Access (Just-In-Time)

Just-In-Time (JIT) database access means no standing access. You raise a request only when needed, get time-bound access after approvals, and access is revoked automatically on expiry.

Approval Matrix

  • Non-Prod Read: Self approval
  • Non-Prod Write: Reporting Manager approval
  • Prod Non-PII Read: Reporting Manager approval
  • Prod Non-PII Write: Reporting Manager approval
  • Prod PII Read: Reporting Manager and SecOps Lead approval
  • Prod PII Write: Denied by default
  • Untagged RDS: Request blocked until tags are added

    Access Governance Controls

    Use this page for the policy controls that directly affect runtime behavior. Duplicate directory listings and placeholder builders have been removed.

    Environment Tags & Organization Scope

    Live

    Tag roots, OUs, and accounts as prod, nonprod, or sandbox. These values are saved by the backend and used by approval workflows and guardrails.

    • Environment tagging is managed only from the AWS Identity Center hierarchy.
    • Approval routing uses these tags to decide the execution plane and matching workflow.
    • Requests to untagged RDS targets stay blocked until classification and environment tagging are complete.

    Approval Workflows

    Required

    Database requests do not proceed unless a matching approval workflow exists for the request context.

    • Match criteria include service type, environment, account, data classification, and access level.
    • Keep one broad nonprod database workflow for controlled testing.
    • Use higher-priority production workflows for sensitive and privileged paths.

    Feature Rollout

    Runtime

    Feature flags decide which user-facing modules and request modes are visible without changing the deployment package.

    • Guided database request mode can be disabled independently.
    • Structured database access remains available as the primary request path.
    • Terminal access, documentation, and support actions stay independently controlled.

    Effective Policy Baseline

    Reference

    The current platform baseline for database access should remain strict and predictable:

    • No standing database access. All database access is just-in-time and time-bound.
    • Sensitive production databases are controlled by approval workflows and write guardrails.
    • Database write exceptions are explicit and tied to named users or groups only.
    • Identity Center users, groups, and permission sets are managed from the dedicated AWS Identity Center tab.
    Users, groups, and permission sets are intentionally not duplicated here. Use AWS Identity Center for directory and hierarchy operations, and Security → Guardrails for enforcement exceptions.

    Access Guardrails

    Define security guardrails to control what services and operations users can request

    AI-Powered Guardrail Generator

    Describe your security requirements in natural language, and AI will generate appropriate guardrails

    MFA verification required for AI-generated guardrails
    How it works: choose a guardrail tile, configure the rules, then click Save Guardrails. These controls apply to future access requests after save.

    Service Access Restrictions

    Control which AWS services users can request access to

    Guidelines: Sensitive services like KMS (encryption keys) and Secrets Manager contain critical credentials and encryption keys. Restrict access to authorized personnel only.

    Delete Operation Restrictions

    Block delete permissions for specific services to prevent accidental data loss

    Warning: Delete operations are irreversible. Consider blocking delete permissions for production databases, S3 buckets with critical data, and KMS keys.

    Create Operation Restrictions

    Control which services users can create new resources in

    Best Practice: Resource creation should go through proper change management. Block create permissions and route requests to DevOps/Infrastructure team.

    Custom Guardrails

    Define custom rules based on tags, resource names, or specific conditions

    Database Write Action Guardrails

    Block write actions (INSERT, UPDATE, DELETE, CREATE, ALTER, DROP, TRUNCATE) by account and DB instance. Add explicit user/group exceptions where needed.

    Policy: When enabled, users can still request read-only access. Write requests are blocked unless the requester email or group is explicitly allowed below.
    Only these selected write actions will be shown to the exception users/groups. Everyone else sees read-only actions only.

    No database write guardrail rules added yet.

    Database Query Guardrails

    Block risky query patterns at query execution time, even when a user has general SELECT or write access.

    Runtime enforcement: These patterns are checked when the user actually executes SQL through PAM, so access can remain read-only while still blocking SELECT *, GRANT, DROP, and other high-risk statements.

    No database query guardrail rules added yet.

    Help & Feedback

    1. I am not able to place the database request as it is asking for data_classification tag

    Please reach out to the DevOps team to add the required tags. Once the tagging is completed, you can proceed with the request.

    2. I am not able to place the database request as IAM-based authentication is not enabled

    Please contact the DevOps team to enable IAM-based authentication for the database. After it is enabled, retry placing the request.

    3. I am not able to view the RDS instances in the selected account

    Please reach out to the DevOps team to verify that the PAM IAM role has the required permissions to read resources in the account.

    4. I am not able to access the URL through AWS Identity Center

    Please contact the SecOps team (Sahil Thakur) to check for any Netskope-related issues.

    5. I am not able to see the application under “My Applications” in AWS

    Please reach out to the SecOps team (Nikita Prasad or Tejas Jual) for assistance.

    6. I am not able to log in to the AWS URL

    Please contact the IT team to verify your access status, especially if you have already raised a request through ITSM.

    7. How can I get write access to a PII database if the request is blocked?

    By default, write access to PII databases is restricted for all users. You will need approvals from Database Owner, DevOps Team, and SecOps Team. Post approvals, the NPAMx admin will grant an exception to allow the request.

    8. Can I place a request on behalf of vendor users?

    No, this functionality is currently restricted to internal users only.

    9. Can I place a request on behalf of my colleague?

    Yes, you can. However, the request will require your colleague’s Reporting Manager (RM) approval, if defined in the workflow.

    10. Whom should I contact for any issues or challenges with the NPAMx application?

    Please reach out to satish.korra@nykaa.com or Manoj.c@nykaa.com.
